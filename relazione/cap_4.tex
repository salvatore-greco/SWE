Sono stati divisi nei package \texttt{UnitTest} e \texttt{IntegrationTest}. Nel primo si va a testare la classe in isolamento, facendo il mock del database se necessario, come fatto un \texttt{AuthUnitTest}, mentre nel secondo si vanno a chiamare i metodi implementati nei DAO interrogando realmente il database. Per l'elenco di tutti i test case si rimanda alla sezione \hyperref[tabella_test]{Tabella test}.

Per agevolare l'automatizzazione degli integration test è stata scritta una classe \texttt{DBSeeder} che va a creare uno schema \texttt{test}, istanziato mediante il file \texttt{db.sql} e popolato tramite il file \texttt{data.sql}. Gli script \textit{SQL} sono eseguiti tramite la classe \texttt{ResourceDatabasePopulator} del package \texttt{springframework.jdbc}
\begin{code}
    \begin{minted}[breaklines]{java}
    public class DBSeeder {

    private Connection connection;

    public DBSeeder() throws SQLException{
        this.connection = ConnectionManager.getInstance().getConnection();
    }

    public void createTestSchema() {
        try {
            Statement stmt = connection.createStatement();
            stmt.executeUpdate("CREATE SCHEMA IF NOT EXISTS test AUTHORIZATION CURRENT_USER");
        } catch (SQLException e) {
            e.printStackTrace();
        }
    }
    public void deleteTestSchema(){
        try {
            connection.setAutoCommit(true);
            Statement stmt = connection.createStatement();
            stmt.executeUpdate("DROP SCHEMA IF EXISTS test CASCADE");
            stmt.executeUpdate("SET search_path TO public");
        } catch (SQLException e) {
            e.printStackTrace();
        }
    }

    public void initDatabaseTest(String filenameDDL, String filenameData){
        try {
            Statement stmt = connection.createStatement();
            stmt.execute("SET search_path TO test");
        } catch (SQLException e) {
            e.printStackTrace();
            throw new RuntimeException("Database error. Check if schema 'test' exists");
        }
        ResourceDatabasePopulator populator = new ResourceDatabasePopulator();
        populator.addScripts(
                new FileSystemResource(filenameDDL), //file must be in root directory
                new FileSystemResource(filenameData)
        );
        populator.populate(this.connection);
    }
}
    \end{minted}
\caption{Classe DBSeeder}
\end{code}
\noindent Il setup e il teardown sono comuni a tutti gli integration test; essi sono definiti nella classe \texttt{BaseIntegrationTest} che ogni classe nel package estende. Il setup va a creare lo schema \texttt{test}, crea le tabelle e imposta l'autocommit della connessione a falso; dopo ogni test case viene eseguito il teardown, dove viene fatto il rollback della transazione. In caso di fallimento di questa operazione, viene impostato un flag \texttt{teardownFail} a vero, che il metodo \texttt{teardownChecker} eseguito dopo ogni test case controlla, facendo fallire tutti i test dato lo stato inconsistente del database. Infine il metodo \texttt{connectionCloser} cancella lo schema precedentemente creato e chiude la connessione.
\begin{code}
    \begin{minted}[breaklines]{java}
    private static void setTestSchema() throws SQLException {
        Statement stmt = connDAO.createStatement();
        stmt.execute("SET search_path TO test");
    }

    @BeforeAll
    public static void init() {
        try {
            dbSeeder = new DBSeeder();
        } catch (SQLException e) {
            e.printStackTrace();
            fail("Failed to instantiate DBSeeder. Check connection to db");
        }
        dbSeeder.createTestSchema();
        assertDoesNotThrow(() -> dbSeeder.initDatabaseTest("db.sql", "data.sql"));
        try {
            connDAO = ConnectionManager.getInstance().getConnection();
            setTestSchema();
            connDAO.setAutoCommit(false);
        } catch (SQLException e) {
            e.printStackTrace();
            fail("Fail to get connection");
        }
    }

    @AfterEach
    public void teardown() {
        try {
            connDAO.rollback();
        } catch (SQLException e) {
            teardownFail = true;
            e.printStackTrace();
        }
    }

    @BeforeEach
    public void teardownChecker() {
        assumeFalse(teardownFail, "Aborting test. DB state is inconsistent");
    }


    @AfterAll
    public static void connectionCloser(){
        try {
            dbSeeder.deleteTestSchema();
            connDAO.close();
        } catch (SQLException e) {
            e.printStackTrace();
        }
    }        
    \end{minted}
    \caption{metodi della classe BaseIntegrationTest}
\end{code}

\subsection{Criteri di progettazione dei test}
La progettazione dei test è stata guidata dall’obiettivo di verificare in maniera sistematica il corretto funzionamento delle diverse componenti del sistema, sia in isolamento sia in interazione con il database. In particolare, sono stati adottati i seguenti criteri:
\begin{itemize}
    \item \textbf{Copertura dei casi d’uso principali}: per ogni funzionalità è stato testato il comportamento atteso nei casi standard, come la creazione, modifica e cancellazione di entità.
    \item \textbf{Gestione dei casi di errore}: sono stati introdotti test specifici per verificare il comportamento del sistema in condizioni non valide, come input errati, entità inesistenti o operazioni non consentite.
    \item \textbf{Casi limite}: sono stati testati scenari particolari come collezioni vuote, risorse non disponibili o vincoli violati (ad esempio prenotazioni duplicate).
    \item \textbf{Test delle operazioni CRUD}: per tutte le classi del package \texttt{ORM} sono stati verificati i metodi di creazione, lettura, aggiornamento e cancellazione, controllando casi di successo e fallimento.
    \item \textbf{Integrità del sistema}: negli integration test è stato verificato che le operazioni producano gli effetti attesi sul database, controllando che lo stato delle entità sia consistente dopo ogni operazione.
\end{itemize}
Infine, l’elevato numero di test sviluppati è anche motivato dall’assenza di un'interfaccia grafica: tale mancanza è stata compensata attraverso una validazione approfondita delle funzionalità del sistema tramite testing.
\subsection{Unit Test}
\subsubsection{DomainModel}
Per il package \texttt{DomainModel} sono stati effettuati test solo di \texttt{EventRoom}, \texttt{Event}, \texttt{Loan}, \texttt{StudyRoom} e \texttt{Room} in quanto le altre classi sono prive di metodi che non siano getter o setter. Come visto in \ref{subsec:nomenclatura} sono stati usati nomi significativi per i test case pertanto si rimanda a \hyperref[UnitTest]{Tabella Unit test} per un elenco esaustivo dei test case effettuati.
\subsubsection{BusinessLogic}
Per il package \texttt{BusinessLogic} è stata testata una sola classe, \texttt{ConcreteAuthManager}; testare in isolamento anche i controller avrebbe portato a una duplicazione dei test case fra Unit Test e Integration test, pertanto è stato scelto di effettuare solo Integration Test.\\
Per testare \texttt{ConcreteAuthManager} viene fatto il mock dei DAO usati all'interno dei metodi, facendone lo stub per restituire un dato prevedibile a ogni esecuzione dei test, senza fare chiamate al database. Per effettuare il mocking è stato usato \texttt{Mockito}.

\subsection{Integration Test}
Come specificato anche in \ref{subsec:struttura} ogni classe in questo package è sottoclasse di \texttt{BaseIntegrationTest} per compiere setup, teardown e controllo sul teardown.

\subsubsection{Business Logic}
Per la business logic è stato seguito un approccio differente, non è stata scritta una fixture per ogni controller ma sono state divise per operazione da svolgere. Sono presenti:
\begin{itemize}
    \item \textbf{CardIntegrationTest} per testare la parte di richiesta e approvazione delle tessere
    \item \textbf{BookIntegrationTest} per testare la parte di aggiunta, modifica e cancellazione di un libro in catalogo da parte dell'amministratore
    \item \textbf{EventIntegrationTest} per testare la creazione, cancellazione e modifica di un evento da parte del bibliotecario e la prenotazione da parte dell'utente
    \item \textbf{LibraryIntegrationTest} per testare la gestione del budget da parte dell'amministratore della biblioteca
    \item \textbf{LoanIntegrationTest} per testare la richiesta e la registrazione dei prestiti, coinvolgendo sia il bibliotecario che l'utente
    \item \textbf{StudyRoomIntegrationTest} per testare la prenotazione e il rilascio dei posti in aula studio da parte dell'utente
\end{itemize}
Il controller di ogni attore è stato istanziato seguendo la procedura di login tramite \texttt{AuthManager} per avere un esperienza più vicina possibile a quella che potrebbe essere di un utilizzatore del software.\\

\section*{Informativa sull'uso dell'intelligenza artificiale generativa}
Si informa che durante la stesura del codice e la fase di design sono stati usati strumenti di intelligenza artificiale generativa come \texttt{Google Gemini} e \texttt{GitHub Copilot} per assistenza allo sviluppo; il tutto è stato supervisionato, letto, compreso e adattato dai sottoscritti.
In particolare \texttt{Google Gemini} è stato usato in fase di design per trarre spunti da idee di progettazione; ad esempio la marker interface \texttt{ControllerInterface} è stata suggerita dall'AI; dai sottoscritti sono state fatte ricerche per validare il suo inserimento al fine di inserirla nel class diagram. \\
\texttt{GitHub Copilot} è stato usato all'interno dell'IDE, non in modalità agentica, tramite apposita estensione per avere un autocompletamento del codice, esso è stato sempre controllato, validato e modificato dai sottoscritti; ad esempio su \texttt{BookDAOIntegrationTest} è stato fatto un ampio uso di tale estensione. Anch'esso è stato validato e modificato dai sottoscritti.
